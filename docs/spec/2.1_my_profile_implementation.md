# **Card VIII – Roles and Access in Your Plan (PlanOwner)**

**Location:** *My Profile (PlanOwner Record) → Overview Tab → Card VIII*  
**Purpose:** Central control panel where the PlanOwner **assigns roles, defines access scopes, sets timing/triggers, and manages suggestions \+ executor releases** — with zero risk of unauthorized sharing.

---

## **1\) Card Identity**

### **Title**

**Roles and Access in Your Plan**

### **Subtitle**

Control who can view parts of your plan, who can suggest additions, and who can release pre-planned access — always under your approval.

### **Core System Principles (Locked)**

1. **Only you (PlanOwner) can share.**  
2. **No one can edit your plan except you.**  
3. **Contributors can only suggest; you approve before anything becomes visible.**  
4. **Executors can release access only through tasks you predefine — even without seeing the content.**  
5. **Beneficiaries can view/download only what you grant — they cannot share.**

---

## **2\) Card Objectives (What this card must enable)**

The PlanOwner must be able to:

* See **all people** connected to roles (Contributors, Executors, Beneficiaries)  
* Add/remove people and change role assignments  
* Define **scope** of access (where in the plan)  
* Define **timing** (immediate / date / event / post-mortem)  
* Define **constraints** (download yes/no, time limit, etc.)  
* Manage **Suggestions Inbox** (approve/reject)  
* Manage **Executor Release Plans** (task-based deliveries)  
* Manage **Audit trail** (“who saw what / who released what / who suggested what”)

## **6\) Module 3 — Access Rules (Scopes & Triggers)**

### **Purpose**

Define **what part of the plan** someone can access and **when**.

### **Core concept**

Access is granted to targets at these layers:

* Section → Category/Subcategory → Record → Card → Entry  
  and can be applied with:  
* immediate / date / event / post-mortem trigger

### **“Grant Access” builder**

Button: **Grant access**

#### **Step 1 — Choose person & role context**

Select:

* Person  
* Role lens (Executor vs Beneficiary vs Contributor)  
  * because the same person may have multiple roles

#### **Step 2 — Choose Access Type**

* **View access**  
* **Contributor suggestion access**  
* **Release task authority** (Executor only)

#### **Step 3 — Choose Target (Scope Tree)**

A hierarchical tree that mirrors your plan structure.

* The tree is checkbox-based:  
  * check parent → checks all children  
  * uncheck child → parent becomes partially checked  
* Important rule:  
  * If a section/category contains records, records are listed  
  * If it contains no records, record layer does **not** appear

Example in *My Family*:

* My Family  
  * Person A  
  * Person B  
  * Person C  
    * Cards…  
      If you grant access to “My Family”, all three are included.  
      If you grant access to “Person C”, only that record appears for them.

#### **Step 4 — Timing / Trigger**

Choose one:

* **Immediate**  
* **On date** (calendar)  
* **Post-mortem** (platform-defined verification trigger)

#### **Constraints**

Depends on role:

**Beneficiary / Executor**

* Download: Allowed / Not allowed  
* Expiry: Never / X days  
* No-share: always enforced (non-editable rule)

**Contributor**

* Access duration: default 15 days (editable)  
* Download: always disabled  
* Share: always disabled

#### **Step 5 — Review & Save**

Shows:

* Target summary  
* Timing summary  
* Download rule  
* Notes (optional)  
  Then Save.

## **8\) Module 5 — Executor Releases & Tasks (Pre-planned Deliveries)**

### **Purpose**

Let PlanOwner predefine deliveries that **Executor will trigger**, including event-based and post-mortem.

### **Key rule (always stated on module top)**

Executor may release access **even without view permission**, because release is **action-only**.

### **Release Plan object**

Each release plan has:

* Recipient (beneficiary email)  
* Target content (scope tree selection)  
* Trigger:  
  * Date  
  * Event  
  * Post-mortem  
* Assigned executor(s)  
* Delivery method:  
  * email notification \+ secure link  
* Visibility:  
  * executor sees label \+ breadcrumbs, not content

### **Create release plan flow**

Button: **Create release**

Steps:

1. Select beneficiary (or add new)  
2. Select target scope (tree)  
3. Set trigger (date / event / post-mortem)  
4. Assign executor(s) (one or multiple)  
5. Label it (human name shown to executor)  
6. Save

### **What happens after save**

* If trigger is date-based:  
  * task is queued for that date  
* If trigger is event-based:  
  * a pending task exists (with reminders)  
* If post-mortem:  
  * tasks become active when verified

### **Executor Task behavior (system)**

* Status: Pending / Completed / Overdue / Reassigned  
* Reminders: automated  
* If multiple executors assigned:  
  * configurable rules:  
    * first-to-complete  
    * or require two-person confirmation (optional future hardening)

### **PlanOwner controls**

* Cancel release  
* Change recipient  
* Change target  
* Change executor(s)  
* Change timing  
  All changes logged.

### **Task Definitions** 

There are four different task types that can be assigned in the PlanAfter platform:

1. Suggestion Tasks   
2. Post-Mortem Activation Tasks  
3. Beneficiary Access Tasks  
4. Post-Mortem Tasks

**A. Suggestion Tasks (Contributor tasks)**  
Tasks that require action from a **Contributor** while the PlanOwner is alive. These tasks usually involve adding, updating, or improving information inside the platform.

**Examples:**

* Upload a missing document

* Update my cousin’s contact information

**B. Post-Mortem Activation Tasks (Executor tasks)**  
Tasks assigned to the **Executor** that become active **only after the PlanOwner’s death**, focused on activating the plan and enabling access for others.

**Examples:**

* Grant platform access to the pre-approved list of people (Beneficiaries)

**C. Beneficiary Access Tasks (Executor-triggered, event-based)**  
Tasks that the **Executor** is responsible for initiating, but only when a **future real-world event occurs** (the timing isn’t fixed on a calendar). These tasks unlock or deliver specific items to beneficiaries at the right moment.

**Examples:**

* Send this video to my daughter on her wedding day

* Give this document to my grandson when he starts driving

**D. Post-Mortem Tasks (Executor and/or Beneficiary tasks)**  
Tasks that become active **after the PlanOwner’s death** and may be assigned to the **Executor**, the **Beneficiary**, or both—depending on who must complete the action.

**Examples:**

* *(Executor)* Cancel my Netflix and Spotify subscriptions

* *(Beneficiary)* Get this document court-approved to receive my car
