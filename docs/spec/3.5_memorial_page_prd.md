# PlanAfter Memorial Page Creation & Management Product Requirements Document

Aligned with the full Post-Loss Support module, the person-record architecture, the current platform role/access model, and the obituary-parity access rule inside the Memorialization tab.

Document status: consolidated working PRD v2 (updated after product review comments)

Applies to: PlanOwner creation flow, memorial administration, memorial public publishing, and invited-role participation

# **1\. Product Context and Position in PlanAfter**

Memorial Page Creation & Management is part of PlanAfter’s Post-Loss Support pillar. It is not a standalone product and it must not behave like an isolated website builder. It is one of the core post-loss remembrance tools, together with Obituary, and it must remain structurally, emotionally, and operationally aligned with the broader Post-Loss Support environment.

Post-Loss Support is a top-level section in the platform. Its role is to orient users during a sensitive moment, generate person-specific support experiences, and provide practical tools such as memorialization. Inside that module, the Memorial Page feature can be reached through the following product paths:

* Quick Action from the Post-Loss Support directory page  
* Step surfaced inside a person-specific Care Plan  
* Memorialization tab inside the deceased person’s record

The memorial feature must therefore feel like a continuation of the person-based post-loss support journey, not like a separate public website product.

# **2\. Core Product Principle**

Every memorial in PlanAfter is person-based, not abstract. The memorial must always be linked to a specific deceased person record. That person's record remains the canonical source of identity, relationship, status, and supporting data.

* A memorial cannot exist without a linked deceased person record.  
* If no person record exists, the flow must create one before the memorial can be created.  
* If the person is not marked deceased, memorial creation must stop until the status is confirmed and updated.  
* All memorial data created through the flow must be written back into the linked person record, either as canonical record data or as memorial-linked presentation data, depending on the field type.

# **3\. Governance Model**

PlanAfter is a single-author, controlled-distribution system. This governance model applies fully to memorialization.

* There is exactly one authoritative editor: the PlanOwner.  
* All other roles interact only through explicitly granted view access, suggestion access, or task-based release actions.  
* No role other than PlanOwner can independently propagate memorial content, approve memorial changes, or redefine memorial access.

If multiple people contribute to the memorial, their contributions remain non-authoritative until the PlanOwner explicitly approves them.

# **4\. Product Goals**

The feature must support four connected actions:

1. Create a memorial page through a guided setup flow that defines who the memorial is for, how it is presented, how it is stored, and how it is initially shared.  
2. Maintain the memorial page over time as an editable remembrance surface where the PlanOwner can enrich content, review suggestions, manage privacy, and control publishing status.  
3. Support both private/invite-only memorialization inside PlanAfter and public memorial publication on a separate dedicated memorial domain.  
4. Keep all memorial behavior aligned with the broader Post-Loss Support module and the platform’s role and access logic.

# **5\. Scope**

This PRD covers the full memorial lifecycle:

* Access rules and role model  
* Creation flow from all supported entry points  
* Verification and plan-entitlement gates  
* Private, invite-only, and public memorial visibility  
* Public publication domain and URL model  
* Theme/design behavior for Basic versus Premium users  
* Memorial object creation and person-record linkage  
* Post-creation memorial administration  
* Suggestion handling and approvals  
* Sharing, release, download, and notification logic  
* Section-by-section memorial editing behavior, including supported attachment types and limits  
* Audit, traceability, and edge cases

This PRD does not define the obituary builder in full detail. However, it explicitly requires that memorial access from a person record use the same access logic as obituary access.

# **6\. Alignment with the Post-Loss Support Module**

The memorial feature must remain structurally aligned with the broader Post-Loss Support module.

* Post-Loss Support can be entered either in personal-loss mode (PlanOwner during life) or in the activated post-death experience.  
* Each support experience is anchored to a deceased person record.  
* Memorialization is one of the Quick Actions tools in the module and may also appear as a Care Plan step.  
* Active loss plans redirect users into the person record’s Care Plan tab; memorialization is a linked tool within that same person-specific environment.

The memorial tool must feel like a calm, emotionally aware continuation of the post-loss experience rather than a separate public-product surface.

# **7\. Eligibility Requirements**

## 7.1 Authentication

A user must be signed in before entering memorial creation or memorial administration.

## 7.2 Verified-user-only access

Memorial creation and memorial administration are available only to authenticated, verified PlanAfter users.

* Required account status: authenticated and verification\_status \= verified  
* Applies to: creation wizard, memorial admin controls, editing tools, privacy controls, invitation controls, media management, publication controls, and suggestion review controls

If a signed-in user is not verified, the system must block access with a verification gate.

* Title: Verify your account to continue  
* Body: Memorial creation and administration are available only to verified PlanAfter users. Please complete account verification to continue.  
* Primary CTA: Verify my account  
* Secondary CTA: Cancel

After successful verification, the user returns to the exact memorial step they attempted to access. No memorial draft object is created before verification is complete.

## 7.3 Plan entitlement model: Basic versus Premium

A basic version of memorialization is available and functional for Basic PlanAfter users. Premium expands memorial functionality but does not change authentication, verification, or role-access requirements.

| Capability | Basic PlanAfter | Premium PlanAfter |
| :---- | :---- | :---- |
| Create memorial from Quick Actions, Care Plan, or Memorialization tab | Yes | Yes |
| Direct memorial administration by PlanOwner | Yes | Yes |
| Private memorial | Yes | Yes |
| Invite-only memorial | Yes | Yes |
| Public memorial publication on dedicated memorial domain | No | Yes |
| Theme / design choice | No — default Basic memorial design only | Yes — choose from available memorial themes/designs |
| Public memorial URL / live publication on external memorial domain | No | Yes |
| Premium upsell inside memorial admin for gated features | Shown where relevant | Not shown |

## 7.4 Dedicated public memorial domain

When a memorial is made fully public and published live, it must be published on a domain that is separate from PlanAfter. The memorial publication domain uses the RememberedAfter brand rather than the PlanAfter product domain.

* Dedicated public memorial domain: separate from PlanAfter  
* Public URL model for live memorial publication: www.\<name\>.rememberafter.com  
* Public live publication is available for Premium subscribers only  
* Private and invite-only memorials do not require public-domain publication

Memorial administration remains inside PlanAfter. The public memorial URL is a viewing surface, not a substitute for the authenticated PlanAfter admin experience. 

# **8\. Role Model in the Memorial Context**

## 8.1 PlanOwner

The PlanOwner is the sole authority and source of truth for the memorial.

* Create memorial  
* Edit memorial directly  
* Publish, unpublish, archive, and delete memorial  
* Define privacy and sharing  
* Invite people and configure access  
* Approve or reject all memorial suggestions  
* Upload and manage media  
* Control theme, notifications, and memorial settings  
* Revoke access at any time

## 8.2 Contributor

A Contributor is a temporary helper operating in suggestion mode.

* May view only the memorial scope explicitly granted  
* May suggest text updates  
* May upload media as suggestions  
* May submit life stories or memorial content as suggestions  
* Cannot directly edit live memorial content  
* Cannot publish, approve, share, or manage settings  
* Access is scope-limited and time-limited by default

## 8.3 Executor

An Executor is an operator of the PlanOwner’s intent, not a memorial content owner.

* May receive memorial-related activation or release tasks  
* May release memorial access to a Beneficiary if the PlanOwner has predefined such a task  
* May confirm post-mortem or event-based triggers that unlock memorial access  
* May view memorial content only if separate readable access has been explicitly granted  
* May act in suggestion mode only if also assigned Contributor access

The Executor must not be treated as a default memorial administrator, co-owner, or direct editor.

## 8.4 Beneficiary

A Beneficiary is a passive recipient.

* May view memorial content only if explicitly shared  
* May download only if explicitly allowed  
* May receive memorial access exactly timed and exactly scoped  
* Cannot edit memorial content  
* Cannot suggest unless also assigned Contributor role  
* Cannot share content through the platform  
* Cannot manage settings or perform release actions

# **9\. Sharing, Release, and Download Rules**

## 9.1 Sharing

Only the PlanOwner can share memorial content, because sharing in PlanAfter means granting access rights.

## 9.2 Release

Only an Executor can release memorial content, and only through predefined tasks configured by the PlanOwner. The Executor cannot choose recipients or content independently.

## 9.3 Download

* PlanOwner: Yes  
* Contributor: No  
* Executor: Yes only if explicitly allowed  
* Beneficiary: Yes only if explicitly allowed

# **10\. Memorial and Obituary Access Logic from a Person record**

Access to Memorial Page functionality from a person record must follow the same access logic as Obituary functionality. Both features live inside the Memorialization layer of the deceased person record and must use the same role model, the same eligibility checks, the same deceased-status dependency, the same timing logic, the same inheritance rules, and the same controlled-distribution permissions.

* Path: Deceased Person Record \> Memorialization tab  
* The Memorialization tab is available only when the person is marked deceased  
* Obituary and Memorial Page are sibling tools under the same access state by default  
* Any difference between obituary and memorial access may exist only if explicitly configured by the PlanOwner at the object level within the shared Memorialization scope

# **11\. Entry Points and Launch Logic**

* Entry Point A: Post-Loss Support \> Quick Actions \> Memorial Page  
* Entry Point B: Deceased Person Record \> Care Plan \> Memorialization-related step  
* Entry Point C: Deceased Person Record \> Memorialization tab

All entry points must resolve to the same memorial object model, the same role/access evaluation logic, and the same data linkage to the deceased person record.

# **12\. High-Level Creation Flow**

* Step 0 — Access validation  
* Step 1 — Identify or confirm the deceased person  
* Step 2 — Complete memorial person details and memorial setup fields  
* Step 3 — Choose your plan (conditional)  
* Step 4 — Privacy and publishing options  
* Step 5 — Theme and design selection (conditional)  
* Step 6 — Memorial ready state  
* Step 7 — First-run memorial onboarding

# **13\. Step 0 — Access Validation**

Before the memorial wizard opens, the system checks:

* Is the user signed in?  
* Is the user verified?  
* Is the user eligible in the current role context?  
* Is the user the PlanOwner for creation/management actions?  
* What plan entitlement does the user have: Basic or Premium?

Outcomes:

* Not signed in \-\> Sign in  
* Signed in but unverified \-\> Verification gate  
* Verified but not eligible by role \-\> Block with role-based explanation  
* Verified PlanOwner on Basic \-\> Continue to setup; Basic restrictions apply later in the flow and in admin  
* Verified PlanOwner on Premium \-\> Full setup access including public publishing and theme selection

# **14\. Step 1 — Identify or Confirm the Deceased Person**

This step is split into a dedicated identity-matching stage before the rest of the memorial fields are shown.

## 14.1 Step 1A — Required initial fields

* First name  
* Last name  
* Date of birth

The user clicks Next after completing these three fields.

## 14.2 Step 1B — Existing record check

On Next, the system must search for an existing person record using first name, last name, and date of birth as matching parameters.

Possible outcomes:

* Outcome A — Matching person record found  
* Outcome B — No matching person record found

## 14.3 Outcome A — Matching record found

If a matching person record is found, the system checks whether that person is already marked deceased.

* If the person is already marked deceased \-\> continue to Step 2 and pre-fill all available data from the record  
* If the person is not marked deceased \-\> show a confirmation step asking whether that person has deceased

Confirmation modal / step:

* Question: Has this person deceased?  
* Options: Yes / No  
* If user selects No \-\> terminate the process with no memorial created  
* If user selects Yes \-\> update the record status to deceased, then collect all available data for that person and display it in Step 2

## 14.4 Outcome B — No matching record found

If no record is found, the system creates a new person record immediately using the first name, last name, and date of birth entered in Step 1A.

Every field completed in the following steps must be stored in that newly created record.

# **15\. Step 2 — Memorial Person Details and Setup Fields**

Once the record is identified or created, the user sees the next step with pre-filled or blank fields as appropriate.

## 15.1 Fields shown in this step

Displayed fields must include all available known data for the person. The user may confirm, enrich, or complete the remaining fields.

* Middle name  
* Gender  
* Relationship  
* Designation  
* Country

Expandable optional section may include:

* Birth year / month / day  
* Birth city / state / country  
* Death year / month / day  
* Death city / state / country

## 15.2 Data persistence

* If the record already existed, new values completed in this step must update the relevant fields in that record according to the platform’s standard edit/save rules  
* If the record was newly created in Step 1, all values entered in this and later steps must be written into the new record  
* Memorial-specific presentation fields remain stored as memorial data linked to the person record

# **16\. Step 3 — Choose Your Plan (Conditional)**

This step is conditional and is visible only if the user is currently on the Basic plan. If the user is already a Premium subscriber, the system skips this step entirely.

## 16.1 Purpose

To make the user’s current plan state visible, explain the difference between Basic and Premium memorial functionality, and allow an in-flow upgrade without losing progress.

## 16.2 Required content

* Display available plans and a side-by-side comparison relevant to memorialization  
* Clearly mark the user’s current plan as Basic  
* Provide a CTA to switch to Premium  
* Preserve all data already entered in Steps 1–2

## 16.3 Required UX behavior

* If user stays on Basic \-\> continue with Basic memorial functionality  
* If user upgrades to Premium \-\> continue the flow with Premium functionality unlocked

# **17\. Step 4 — Privacy and Publishing Options**

This step defines both the visibility model and, when applicable, the public publication behavior.

## 17.1 Available options by plan

* Basic users: Private; Invite-only  
* Premium users: Private; Invite-only; Public

## 17.2 Basic behavior

Basic users must see only two visibility choices:

* Private memorial page  
* Invite-only memorial page

A Basic user cannot make the page public and cannot publish it live on the external memorial domain.

## 17.3 Premium behavior

Premium users see the same two options plus a third public option.

* Public memorial page

If Public is selected, the UI must expand and show the public publishing settings.

## 17.4 Public publishing settings (Premium only)

* Dedicated public memorial domain under the RememberedAfter brand  
* Public URL pattern: www.\<name\>.rememberafter.com  
* Real-time uniqueness validation for the subdomain / slug component  
* Clear statement that the page will be publicly visible without PlanAfter sign-in once published live

## 17.5 Terms and consent

The user must accept Terms before continuing. The Continue button remains disabled until one privacy option is selected and Terms are accepted.

# **18\. Step 5 — Theme and Design Selection (Conditional)**

Theme and design behavior depends on plan entitlement.

## 18.1 Basic behavior

Basic users do not choose between different memorial themes or designs. The system automatically applies the default Basic memorial design.

* The user should see an informational note that design personalization is not available on Basic  
* The product may either skip the theme chooser entirely or show a locked, single-theme screen; either approach is acceptable as long as the user clearly understands the limitation

## 18.2 Premium behavior

Premium users can choose from the available memorial themes/designs.

* One theme selection required before the memorial is finalized  
* Theme can be changed later in memorial administration  
* Theme changes must not remove content or change permissions

# **19\. Step 6 — Memorial Ready State**

When the final setup step is completed, the system must:

* Create the memorial object  
* Link it to the deceased person record  
* Persist privacy mode and memorial settings  
* Persist public domain settings if public publication was selected and the user is Premium  
* Persist theme or default Basic design assignment  
* Assign the current user as PlanOwner of the memorial surface  
* Log creation event

The user then sees a confirmation state and is routed into memorial administration.

# **20\. Step 7 — First-Run Memorial Onboarding**

The first-run onboarding should orient the PlanOwner inside the memorial administration experience.

* Edit page  
* Invite people  
* Add cover picture  
* Change theme / design  
* Privacy and publishing  
* Suggestion review  
* Notifications  
* Admin settings

The onboarding must be skippable and replayable later.

# **21\. Memorial Administration Experience**

After creation, the memorial becomes an ongoing management surface. This is not a multi-editor memorial CMS. It is a PlanOwner-controlled memorial workspace aligned with the broader Post-Loss Support architecture.

It must support:

* Direct PlanOwner editing  
* Suggestion intake from Contributors where allowed  
* Scoped viewing by Beneficiaries  
* Task/release workflows by Executors where configured  
* Audit-safe changes  
* Continuing memorial enrichment over time  
* Public publication controls for Premium users only

# **22\. Memorial Tabs and Post-Creation Structure**

The memorial page uses a persistent tab structure. Each tab must have a clear editing model, a clear role behavior model, and explicit rules for attachments where attachments are supported.

* About  
* Life  
* Gallery  
* Stories  
* Admin

# **23\. About Tab**

## 23.1 Purpose

The About tab is the memorial landing tab and the main remembrance surface. It introduces the deceased person and provides the core memorial text, dedication line, and lightweight remembrance interactions.

## 23.2 Editing model

The PlanOwner sees a text editor block for the main memorial text. The editor must allow writing, formatting, and editing the memorial description.

* Editor type: rich-text editor  
* Supported formatting: bold, italic, underline, quote, bullet list, numbered list, hyperlink, paragraph styles  
* Primary actions: Save draft, Publish changes, Cancel

## 23.3 Tribute area

The tab may include symbolic tribute actions such as Lay a Flower, Light a Candle, and Leave a Note. Any tribute or note submitted by a non-PlanOwner must be treated as a submission or suggestion, never as a direct live edit.

## 23.4 Attachments

No general record attachment uploader is required in the About tab in v1. If tribute-note attachments are introduced later, they must follow the same attachment standards as Stories.

## 23.5 Role behavior

* PlanOwner: can directly edit all About content  
* Contributor: may suggest changes if scoped access exists  
* Beneficiary: may view if shared and may interact with lightweight tribute submission only if enabled and modeled as a submission rather than an edit  
* Executor: may view only if separately granted and may handle releases/tasks related to memorial delivery

# **24\. Life Tab**

## 24.1 Purpose

The Life tab is the structured chapter-by-chapter biography of the deceased. It is intended for a more curated narrative about milestones, life periods, values, and achievements.

## 24.2 Editor structure

The core unit is a Life Story Chapter editor box.

* Fields: Title (optional), Chapter body (required)  
* Editor type: rich-text editor  
* Supported formatting: bold, italic, underline, quote, bullet list, numbered list, hyperlink, divider, paragraph styles  
* Primary actions: Save draft, Publish chapter, Edit chapter, Delete chapter, Reorder chapter

## 24.3 Supported attachments and limits

* Images: JPG, JPEG, PNG, WEBP — max 10 MB per record  
* Documents: PDF, DOCX, TXT — max 25 MB per record  
* Audio: MP3, WAV, M4A — max 50 MB per record  
* Video: MP4, MOV — max 100 MB per record

## 24.4 Role behavior

* PlanOwner: creates, edits, reorders, publishes, and unpublishes chapters directly  
* Contributor: may submit chapter suggestions or suggested edits  
* Beneficiary: read-only if shared  
* Executor: no default editorial control

# **25\. Gallery Tab**

## 25.1 Purpose

The Gallery tab is the media display and management surface of the memorial.

## 25.2 Structure

The Gallery tab should support separate internal surfaces for Photos, Videos, and Audio. The PlanOwner must be able to upload, review, caption, and remove media items.

## 25.3 Upload component

* Upload modes: choose records or drag and drop  
* Optional source integrations may be added later, but local upload is required in v1  
* Each item may support title, caption, and visibility controls

## 25.4 Supported formats and limits

* Photos: JPG, JPEG, PNG, WEBP — max 10 MB per record  
* Videos: MP4, MOV — max 100 MB per record  
* Audio: MP3, WAV, M4A — max 50 MB per record

## 25.5 Role behavior

* PlanOwner: full media upload and management  
* Contributor: may upload media as suggestions if allowed; uploads are not authoritative until approved  
* Beneficiary: view-only if shared; download only if allowed  
* Executor: no default gallery administration

## 25.6 Plan-based behavior

* Basic: gallery works, but no public publishing on the dedicated memorial domain and no advanced theme-dependent presentation controls  
* Premium: gallery works and can be part of a publicly published memorial page; premium-only enhancements may be surfaced here if required

# **26\. Stories Tab**

## 26.1 Purpose

The Stories tab is the personal memory-sharing area. It is different from Life: Life is a curated biography; Stories is a place for personal narratives and recollections.

## 26.2 Editor structure

The core unit is a Story editor box.

* Fields: Title (optional), Story body (required), Author attribution  
* Editor type: rich-text editor  
* Supported formatting: bold, italic, underline, quote, bullet list, numbered list, hyperlink, paragraph styles  
* Primary actions: Save draft, Publish story, Edit story, Hide story, Remove story

## 26.3 Supported attachments and limits

* Images: JPG, JPEG, PNG, WEBP — max 10 MB per record  
* Documents: PDF, DOCX, TXT — max 25 MB per record  
* Audio: MP3, WAV, M4A — max 50 MB per record  
* Video: MP4, MOV — max 100 MB per record

## 26.4 Role behavior

* PlanOwner: can publish directly and can edit, hide, feature, and remove stories  
* Contributor: may submit stories as suggestions  
* Beneficiary: may submit a passive story entry only if enabled by the PlanOwner and defined as a non-authoritative submission  
* Executor: no default editorial authority

# **27\. Admin Tab / Admin Menu**

All authoritative controls in the memorial admin surface are PlanOwner-only.

## 27.1 PlanOwner-only controls

* Edit page  
* Add or replace cover picture  
* Change theme / design  
* Manage memorial privacy and public publishing status  
* Invite people  
* Configure access  
* Review and approve suggestions  
* Notification settings  
* Archive, delete, and unpublish

## 27.2 Premium-only admin controls

* Make memorial public  
* Publish live on the dedicated memorial domain  
* Choose between available themes/designs  
* Change public memorial URL / subdomain within allowed rules

## 27.3 Basic-plan admin behavior

Basic users may fully manage the memorial within the limits of the Basic plan, but premium-gated controls must either be hidden or shown as locked with an upgrade CTA.

# **28\. Invitation Flow**

Only the PlanOwner may invite people to memorial access.

Invite flow fields:

* Contact/person selector  
* Optional create new person  
* Role assignment  
* Access level  
* Scope  
* Timing  
* Notes

This must reuse the platform’s existing role/access logic rather than creating a memorial-only invitation model.

# **29\. Suggestion Flow**

All non-PlanOwner memorial changes are suggestions only.

Applies to:

* Memorial text suggestions  
* Tribute/story suggestions  
* Life chapter suggestions  
* Media upload suggestions  
* Cover image suggestions

Only the PlanOwner can approve or reject suggestions. Rejected suggestions must remain retained for audit visibility to the PlanOwner.

# **30\. Event-Based and Post-Mortem Memorial Release**

The memorial PRD must support memorial-related release logic consistent with the broader platform.

* Release memorial access post-mortem  
* Release memorial videos or letters on a specific life event  
* Deliver memorial content to a Beneficiary through Executor-triggered event confirmation

Critical rule: an Executor may release memorial content without seeing the content itself. Release permission is independent from view permission.

# **31\. Notifications**

The memorial can include notification preferences, but they must align with roles.

* PlanOwner: full memorial notification control  
* Contributor: only notifications related to assigned suggestion tasks or suggestion outcomes  
* Beneficiary: only notifications related to memorial access or shared memorial events if enabled  
* Executor: only task and release notifications relevant to assigned memorial tasks

# **32\. Data Integration with the Person record**

The deceased person record remains the canonical identity source. The memorial stores presentation-layer content, theme/design assignment, privacy settings, public-publishing settings, media references, stories, and revision history.

* If shared identity fields are edited inside the memorial, the system must ask whether to update memorial only or memorial plus person record  
* Visitor stories, tributes, or contributor submissions must not overwrite canonical person-record data automatically  
* If no record existed at Step 1, the system-created record becomes the authoritative linked record for all memorial data captured later in the flow

# **33\. Access Hierarchy and Inheritance**

The memorial is treated as an object within the platform’s hierarchical access system.

* Existence visibility: does the memorial or memorial tab exist in the user’s UI?  
* Readable access: can the user open it?  
* Action access: can the user suggest, release, or download?

Inheritance rules:

* Higher-level access inherits downward  
* More specific rule overrides less specific  
* Deny overrides allow  
* Release permission is independent from view permission

# **34\. Audit, Proof, and Legal Traceability**

The memorial feature must participate in the platform’s immutable logging model.

* Memorial creation  
* Memorial publication state changes  
* Access delivery  
* Suggestion approval or rejection  
* Contributor attribution  
* Executor release actions  
* Task completion, non-completion, reassignment, or expiry where memorial access depends on an Executor task

# **35\. Edge Cases**

* User abandons flow before verification: no memorial draft created  
* User abandons flow at plan step or billing: setup data preserved; memorial not created yet  
* Person not marked deceased and user answers No to deceased confirmation: process terminates  
* Existing obituary exists but memorial does not: obituary and memorial remain separate objects under the same Memorialization access logic  
* Contributor access expires mid-suggestion: pending suggestions remain in audit trail; contributor loses visibility  
* Executor can no longer serve: memorial-related release tasks can be reassigned according to executor-failure logic  
* Basic user later upgrades to Premium: existing memorial keeps content and gains public publishing and theme-choice functionality

# 36\. Acceptance Criteria

* Memorial creation is available from Quick Actions, Care Plan, and the Memorialization tab  
* Every memorial is linked to a deceased person record  
* Memorial and obituary use the same access logic from a person record  
* Memorial creation and administration are blocked for unverified users  
* Basic PlanAfter users can create and manage a private or invite-only memorial  
* Premium users can additionally make the memorial public and publish it live on the dedicated memorial domain  
* Basic users cannot choose between different themes/designs; Premium users can  
* The public publication URL model follows the required dedicated memorial-domain pattern  
* Only PlanOwner can directly create, edit, publish, approve, reject, share, archive, or delete memorial content  
* Contributors can only interact in suggestion mode  
* Executors can perform memorial-related release actions without needing content visibility  
* Beneficiaries are read-only unless download is explicitly allowed  
* The detailed step flow uses the identity-check logic requested in Step 1  
* The Choose Your Plan step is conditional and shown only to Basic users  
* Privacy options are plan-aware: Basic \= private/invite-only; Premium \= private/invite-only/public  
* About, Life, Gallery, Stories, and Admin tabs exist and follow the role model  
* Where attachment functionality exists, supported formats and size limits are explicitly defined in the PRD
