## 2.2.1 — Purpose
Family Member Record is a dedicated, secure space for everything related to people in PlanOwner's life. Backbone of:
- Family Tree visualization
- Inheritance simulations & legal heirs logic
- Will & Legal document generation
- Executor workflows
- Guardianship planning
- Post-Loss Support actions
- Emotional Legacy & Life Story
- Shared household logic

Every Family Member is a **single, unique Person Record** used throughout the entire system.

## 2.2.2 — Creation Methods
1. Automatic during Onboarding (placeholder records: "Child 1", "Child 2", empty Partner, Mother/Father)
2. Manual via Add Family button (Header), Family & Relationships card, My Family page, Family Tree, or Link Record flow
3. Auto-suggestion (uploads child medical doc → "Create record?", guardian named in Will → "Create Family Member?", etc.)

**Hierarchy rule:** Grandparent / Grandchild / Extended Family can only be added AFTER the connecting parent exists. When adding a child, BOTH parents must be specified (placeholder "Unknown" is acceptable).

## 2.2.3 — Three-Layer Relationship Architecture (CRITICAL)

| Layer | What | When visible |
|-------|------|--------------|
| **Layer 1 — Relationship Type** | System-level classification (Parent, Child, Sibling, Partner / Spouse, etc.) | Edit mode only |
| **Layer 2 — Specific Role** | User-facing, human-readable role (Mother, Daughter, Aunt, Husband) — always **relative to the currently viewed record** | Always |
| **Layer 3 — Timeline Conditional Dates** | Optional dates describing when a relationship started/ended | When relevant |

**Perspective rule:** When same person is viewed from a different family member's record, system **derives** the correct role for that perspective. Derived roles are NEVER stored — computed dynamically. *Example: Maria stored as "Daughter" from grandma's record → from PlanOwner's record automatically shows as "Aunt".*

## 2.2.3.2 — Relationship Engine System Rules

### 1. Bidirectional always
When user creates relationship A→B, system auto-creates B→A. Same dates stored on the link, identical from both sides. Wording differs based on perspective (Layer 2 derived).

### 2. Recalculation triggers
After any relationship edit, run Relationship Recalculation:
- **Tier A (auto-create, no confirmation):** logically certain (Current Husband ↔ Current Wife)
- **Tier B (auto-suggest, requires confirmation):** plausible (Child ↔ Parent — could be biological/step/adoptive, ask)

### 3. Children: spouse-aware capture, never assumed
When anchor adds a child AND anchor has a spouse → child also recorded under spouse's Children, but PROMPT: "What is [spouse]'s relationship to this child?" (Biological / Adoptive / Step). Until answered, treated as "needs clarification".

### 4. Siblings: generic first, confirm type later
"Add sibling" → create immediate two-way link with generic "Sibling" type. When user views the sibling's record → prompt: "Biological / Half / Step?" Until confirmed: "Sibling — needs clarification".

### 5. Step / In-Law are suggested, not assumed
Marriage-derived relationships easy to misclassify — system may suggest, never auto-finalize.

### 6. Conflict handling + upgrades
- Impossible state (e.g. two current spouses) → BLOCK + require user choice
- More info available → upgrade specificity:
  - Sibling: unknown → Half → Full (as parents added)
  - Parent: Biological Parent → Biological Mother/Father (as gender becomes known)

### Core Rules (LOCKED)
- **No summary card owns data**
- **All changes propagate from Record Input Cards**
- **Actions on summary cards trigger controlled flows** (open record, modal, etc.)
- **Layer 2 (Specific Role) is always the visible truth**
- **Summary cards are permission-aware**
- **Same Record → same data → everywhere**

### Record Header Summary Card — Elements
- Back Button (system nav)

## Family Tree visualization

Family Tree is a **PlanOwner-anchored view** of the family graph. PlanOwner ("Me") in middle.

- Children appear ABOVE PlanOwner
- Parents appear BELOW
- Grandparents further below
- Partner alongside

### Tree node = Linked Summary Card
- Photo / avatar
- Name
- Specific Role (Layer 2)
- Key date line (DoB / DoD)
- Age in brackets

### Quick Add (from PlanOwner anchor only)
- Add Parent / Add Child / Add Sibling / Add Partner / Skip for Later
- Quick Add ONLY supports relatives directly linked to PlanOwner

### Visual relationship lines (REQUIRED behavior)
- **Solid lines** = biological
- **Dashed lines** = step
- **Distinct style** = half
- **On hover**: line/path to PlanOwner highlights in different color

### Three view modes (tabs)
1. **Family Tree** — visual generational layout
2. **Family Members** — structured list (PlanOwner / Spouse / Former Spouse/Partners / Children / Grandparents / Extended Family / Pets)
3. **Timeline** — narrative milestones ("Grandparents born" / "Parents born" / "My birth" / "Children born")

Plus "My Family" quick-access strip — circular avatars at top for fast jumping.
